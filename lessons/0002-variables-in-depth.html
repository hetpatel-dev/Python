Lesson 2: Variables in Depth

concept Variables

In Lesson 1 you used variables as "named boxes." Now we'll open the box and see what's really going on. Understanding variables deeply is the single most important thing for avoiding bugs in Python.

1. Variables Are Names, Not Boxes

Most programming languages teach variables as containers that hold values. Python variables are not containers — they are labels (names) that reference objects.

a = [1, 2, 3]
b = a       # b does NOT get a copy. b is another label for THE SAME list.
b.append(4)
print(a)    # [1, 2, 3, 4] — a changed too!
Think of it this way: Objects live in memory. Variables are just Post-it notes stuck to those objects. You can stick multiple notes on the same object.

This is the Python data model (official docs): every object has an identity (memory address), a type, and a value. The variable stores the identity, not the object itself.

2. Identity: id() and is

Every object has a unique identity that never changes. Use id() to see it:

x = 42
y = 42
print(id(x))   # 140735232337832 (some integer)
print(id(y))   # 140735232337832 (SAME — Python reused the integer object)

a = [1, 2]
b = [1, 2]
print(id(a))   # 2345678901234
print(id(b))   # 2345678905678 (DIFFERENT — lists are mutable, always new)

is compares identities. == compares values.

a = [1, 2]
b = [1, 2]
c = a

print(a == b)   # True (same value)
print(a is b)   # False (different objects)
print(a is c)   # True (same object, just different name)

From Python docs: "An object's identity never changes once it has been created; you may think of it as the object's address in memory."

3. Mutable vs Immutable Types

Some objects can change their value (mutable), others can't (immutable). This is critical for understanding how variables behave.

MutableImmutable
listint
dictfloat
setstr
bytearraytuple
custom class instancesbool
frozenset
bytes
# Immutable: "changing" creates a NEW object
x = 10
print(id(x))   # 140735232337...
x = x + 1      # This creates a NEW int object
print(id(x))   # Different! x now labels a new object

# Mutable: changing modifies the SAME object
items = [1, 2, 3]
print(id(items))   # 234567...
items.append(4)    # Modifies the list IN PLACE
print(id(items))   # SAME! The object is unchanged, only its value changed
Key insight: For immutable types, x = x + 1 reassigns x to a new object. For mutable types, x.append(1) modifies the existing object that x refers to.

4. Assignment Patterns

Basic assignment

x = 5  # binds name x to object 5

Multiple assignment

a, b, c = 1, 2, 3   # a=1, b=2, c=3
x = y = z = 0        # all three reference the same 0 object

Swapping (Pythonic!)

a, b = b, a  # swap — evaluates right side first, then assigns

From the docs: "the right-hand side expressions are evaluated first before any of the assignments take place." This is why swapping works.

Tuple unpacking

coordinates = (10, 20)
x, y = coordinates   # x=10, y=20

# Works with any iterable
name, age = ["Alice", 30]
first, *rest = [1, 2, 3, 4]   # first=1, rest=[2, 3, 4]   (PEP 3132)
*head, last = [1, 2, 3, 4]    # head=[1, 2, 3], last=4

5. Augmented Assignment (+=, -=, etc.)

These combine an operation with assignment. But they behave differently for mutable vs immutable!

# With immutable types: creates a new object
x = 10
x += 5   # equivalent to x = x + 5 → new object

# With mutable types: may modify in-place
list1 = [1, 2]
list2 = list1
list1 += [3, 4]      # IN-PLACE: modifies the list, list2 also changes
print(list2)          # [1, 2, 3, 4]

# Compare with + (always creates new object)
list1 = [1, 2]
list2 = list1
list1 = list1 + [3, 4]  # CREATES NEW: list2 unchanged
print(list2)            # [1, 2]

From the docs: "when possible, the actual operation is performed in-place, meaning that rather than creating a new object and assigning that to the target, the old object is modified instead."

6. Type Annotations (Type Hints)

Python is dynamically typed, but you can annotate variable types. These are ignored at runtime but checked by tools like mypy.

name: str = "Alice"
age: int = 30
height: float = 1.75
is_active: bool = True

# Without value — annotation only
nickname: str  # no value yet, but we've documented the intended type

This is PEP 526. Use type hints for clarity — they make your code self-documenting and catch bugs early.

7. Naming Rules and Conventions (PEP 8)

  • Allowed characters: letters, digits, underscores — but can't start with a digit
  • Case-sensitive: name and Name are different
  • Keywords are reserved: if, for, while, class, etc. cannot be variable names
  • Convention: snake_case for variables, UPPER_CASE for constants
  • Private by convention: _leading_underscore means "internal use"
  • Dunder (double underscore): __name__ — Python special methods, don't invent your own

8. del — Deleting a Variable

del removes the name (not the object). The object may still exist if other names reference it.

x = [1, 2, 3]
y = x
del x           # removes the name x
print(y)        # [1, 2, 3] — object still exists via y
# print(x)      # NameError: name 'x' is not defined

9. Variable Scope (Brief Introduction)

Variables defined inside a function are local to that function. Variables at the top level are global.

x = 10          # global

def func():
    y = 5       # local to func
    print(x)    # can READ global
    print(y)

func()
# print(y)      # NameError — y is not defined here

We'll explore scope deeply in a future lesson on functions.
